72 /*
73  * (true as of 2001-9-24)
74  * Thread-safety note: changes to these flags are not atomic, so
75  * the only thing preventing lost updates is the timing of the changes.
76  *
77  * As long as the following are isolated from each other for any one class,
78  * nearly all flag updates will be safe:
79  * - compile-time
80  * - loading in one thread (not including +load) without messaging
81  * - initializing in one thread with messaging from that thread only
82  * - multi-threaded messaging with method caching
83  *
84  * The current code doesn't protect loading yet.
85  *
86  * Times when the flags may change:
87  * CLS_CLASS: compile-time, hand-built classes
88  * CLS_META: compile time, hand-built classes
89  * CLS_INITIALIZED: initialize
90  * CLS_POSING: unsafe, but posing has other thread-safety problems
91  * CLS_MAPPED: compile-time
92  * CLS_FLUSH_CACHE: messaging
93  * CLS_GROW_CACHE: messaging
94  * FLUSH_CACHE and GROW_CACHE are protected from each other by the
95  * cacheUpdateLock.
96  * CLS_NEED_BIND: load, initialize
97  * CLS_METHOD_ARRAY: load
98  * CLS_JAVA_HYBRID: hand-built classes
99  * CLS_JAVA_CLASS: hand-built classes, initialize
100  * CLS_INITIALIZING: initialize
101  *
102  * The only unsafe updates are:
103  * - posing (unsafe anyway)
104  * - hand-built classes (including JavaBridge classes)
105  * There is a short time between objc_addClass inserts the new class
106  * into the class_hash and the builder setting the right flags.
107  * A thread looking at the class_hash could send a message to the class
108  * and trigger initialization, and the changes to the initialization
109  * flags and the hand-adjusted flags could collide.
110  * Solution: don't do that.
111  */
